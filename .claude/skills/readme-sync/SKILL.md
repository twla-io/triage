# README sync (from Domain.hs)

`src/Domain.hs` is the specification (see `CLAUDE.md`). README.md's
domain-model section is a generated view of that spec for humans landing on
the repo — it must never drift from the actual types, or it becomes exactly
the kind of stale, misleading doc this repo is trying to avoid.

## Hard rule: read the file, don't recall it

Before writing anything, `view`/read the full current `src/Domain.hs`. Every
type name, constructor, field, and function signature that goes into
README.md must come from that read, not from memory of a previous version,
not from `docs/domain-model.md`, not from chat history. If
`docs/domain-model.md` disagrees with what you just read in `Domain.hs`,
trust `Domain.hs` and flag the disagreement to the user — don't silently
resolve it in either direction.

If something about the code is genuinely ambiguous (e.g. you can't tell from
the type alone why a constructor is exported openly vs. sealed), say so in
the output as a TODO rather than inventing a plausible-sounding rationale.

## Scope: only the generated section

README.md contains a marked, generated region:

```markdown
<!-- DOMAIN-MODEL:START (generated by readme-sync — do not hand-edit) -->
...
<!-- DOMAIN-MODEL:END -->
```

Only ever replace the content between these markers. Everything outside them
(project description, setup instructions, badges, license, contribution
guide) is hand-written — leave it byte-for-byte untouched. If the markers
don't exist yet in README.md, ask the user where to insert them rather than
guessing a location; don't restructure an existing README to make room.

## What goes in the generated section

Derive these directly from the current `src/Domain.hs`, in this order:

1. **One-paragraph purpose** — what the module specifies (copy the intent
   from its own header comment if present; don't paraphrase it into
   something it doesn't say).
2. **Core types** — each top-level `data`/`newtype` with exported
   constructors, one line each: name, and what it represents in plain
   language derived from field names and any adjacent comments. Group by
   the module's own section dividers (the `═══` banner comments) if present
   — those are the author's intended groupings, use them rather than
   inventing your own.
3. **Sealed vs. open** — a short list of which types hide their constructors
   (check the export list at the top of the file for the presence/absence
   of `(..)`) and, for each sealed one, which smart constructor to use
   instead. Don't guess at "why" beyond what's stated in a comment near the
   type — if there's no comment, just state the fact (sealed / open) without
   inventing a justification.
4. **Key functions** — the protocol-level functions exported from the
   bottom section of the file (matching/booking/waitlist logic), with their
   real type signatures, verbatim.
5. **Codegen pointer** — if the file's header comment references codegen
   skills (e.g. `triage-db-codegen`), reproduce that pointer so README
   readers know other layers are generated from this file, not hand-written.

## After generating

- Show the user a diff of the generated section before writing it, unless
  they've asked you to just proceed.
- Do not touch `docs/domain-model.md`, `docs/decisions.md`, or
  `docs/modeling-principles.md` as part of this skill — those are separate,
  hand-curated documents. If this run surfaces a conflict between them and
  the code, name the conflict and let the user decide whether to update
  those files too; don't do it silently as a side effect of a README sync.

## Optional: trigger automatically after Domain.hs edits

This skill is conversationally triggered — it won't run itself just because
`Domain.hs` changed on disk. To get an automatic reminder, add a Stop hook
that checks whether `src/Domain.hs` was modified in the session and, if so,
prompts to run this skill before ending. See Claude Code's hooks
documentation for the exact hook config; this skill doesn't assume one is
installed.
